packman-project-file(5) Manual Page

NAME

project file - A complete specification of the packman project file and how to write one

SYNOPSIS

A project will typically use one or more packman project files in conjunction with the pull command.

DESCRIPTION

A project file is a text file following the XML document specification. It’s probably best to start with an example:

<project toolsVersion="5.14">
   <!-- This is a comment -->
   <dependency name="cmake">
      <package name="cmake-x64" version="3.7.0" /> <!--this one is the default -->
      <package name="cmake-linux-x86_64" version="3.7.2" platforms="linux"/>
   </dependency>
   <dependency name="GoogleTest">
      <package name="googletest-nv-${platform}" version="1.4.0.5" platforms="linux win.vc140 mac"/>
      <source path="googletest/ps4" platforms="ps4" />
   </dependency>
</project>

In summary, this document describes a project that has two dependencies: cmake and GoogleTest. In the case of cmake the dependency is being fulfilled with different packages depending on the platform. If linux platform is specified the package named cmake-linux-x86_64 at version 3.7.2 will be used. For all other platforms, including when no platform is specified with pull command, the cmake-x64 package at version 3.7.0 will be used. The GoogleTest dependency is being fulfilled with a package for the platforms linux, win.vc140, and mac. The package name is different for each platform but the naming follows a template that is used above. On the ps4 platform the dependency is being fulfilled with source code that already resides at the path specified.

Note
 All paths in packman XML files follow Unix convention and use forward slashes.

Please read the sections below for details on each element in the XML document.

project

A valid document must start with the project element and specify the minimum toolsVersion that is required (this corresponds to packman version). The software has been backwards compatible with earlier versions of the XML schema but when new features are added to the XML schema the user must specify the minimum version of packman that is needed to support the new feature. That way the user can be informed when an attempt is made to parse an XML document that is too new for the version of packman being used. For instance, previous to 3.0 we only supported GTL for remote storage but then added S3 as an option. The features added to the XML schema to support this option therefore required the user to specify toolsVersion 3.0 or higher.

With version 5.0 we have expanded this feature significantly. With this version we have introduced the specification of any remote server. We currently support any HTTP(s) server, S3 bucket or GTL account as the remote server for packages. We also support a cascade of remotes, i.e. a list of fallback remote servers can be specified if package is not found or accessible on the primary server(s). This is simply done by specifying a list of remote server names at the top of your project file. If we wanted to search first on a remote server named http-packages and fall back to GTL service account that has been configured as remote server named gtl we would do it like this:

<project toolsVersion="5.14" remotes="http-packages gtl">

at the top of your document. If you specify a toolsVersion lower than 5.6 you cannot use the remotes attribute or the remote elements. The remotes attribute can also be specified on a package and will override the project level setting for that package.

Note
 You can only reference remotes that are defined inside the project file. If you refer to a remote that is not defined in the file the parser will issue an error. See below on how to define a remote.

See remote-order topic for how the remotes setting from different sources in packman interact.

remote

The XML document can contain zero or more remote elements. This element is used to configure a remote server for packages. This element requires two attributes to be set: name and type. The name is used to reference the remote through the remotes attribute that is available for use in the project/config root elements but also available on the package element in project files. This name can also be used on the command line with the --remote option (see for instance publish). The type attribute currently supports the following values:

  • gtl

  • s3

  • http

  • https

Some remotes require the attribute packageLocation to be set as well. The value for this attribute depends on the type of service being configured. For s3 remotes the packageLocation is simply the name of the S3 bucket. For http(s) remotes the packageLocation is a URL, usually employing some variable substitution. For example this could be used to utilize GitHub as a package server:

<remote name="github" type="https"
  packageLocation="github/NVIDIAGameWorks/${name}/releases/download/${version}/${name}-${version}.zip" />

Notice how the values of the attributes name and version from the package element (see project file) can be substituted into the URL. If your http(s) remote contains some packages in 7z containers and other packages in zip containers then omit the file extension and packman will search for both.

You can also get quite crafty with URL construction and attribute substitution. For instance, this is a way configure a GitLab host, with repos under the group carbon/nvtools as a remote server:

<remote name="gitlab-nvtools" type="https"
  packageLocation="gitlab-master.nvidia.com/api/v4/projects/carbon%2Fnvtools%2F${name}/repository/archive.zip?sha=${version}" />

The package name in this context is the repo name and the version is the commit hash:

<package name="nvfilecopy" version="73a13bd22a433cb61c8619132854ee612e45fd72"/>

This will deliver the source from the repo which is useful for languages that can execute source directly, like for instance Python.

For improved performance when using S3 buckets it’s common to create an AWS CloudFront distribution in front of the bucket. This can all be configured using the AWS console. Once that is done the upload and download are asymmetric. For example, the packages-for-cloudfront bucket has a matching CloudFront distribution and the download is performed via http/https like this:

<remote name="cloudfront" type="https"
    packageLocation="d4i3qtqj3r0z5.cloudfront.net/${name}@${version}" />

Notice that the upload will be performed via a remote of type s3 with credentials that allow writing to the bucket.

Note
 CloudFront can only serve packages that are public on S3. To streamline this process all packages uploaded to packages-for-cloudfront bucket on S3 are automatically public.

credentials

Some remotes require authentication. A remote can therefore contain a credentials element. This element is used to configure access credentials for that specific remote server. Here is how we configure the packman s3 bucket for read access as a remote:

<remote name="s3" type="s3" packageLocation="packman">
   <credentials id="AKIAJHPSPBMWMTZS6TJA" key="vK3d0lHiQjEW9krFfvKA4OLpuHGxi2L4/Q4r4IuT"
      errorUrl="https://gitlab-master.nvidia.com/hfannar/packman/wikis/credentials/failure-using-s3-packman-read-access"/>
</remote>

The id and key attributes are of course required and they need to provide read access to the remote server’s packageLocation. When configuring remotes for upload the credentials must also allow writing - but that type of configuration would normally not be found in a project XML file. That type of configuration is normally stored in separate config files (see config file). The optional errorUrl attribute is used to specify a web page that packman can redirect to if the access credentials are not accepted by the remote server. This is helpful to aid users when:

  1. credentials have been changed (key rotation) - the latest credentials are published on an internally accessible web server and the address can be specified in errorUrl. Note that key rotation is typically only performed for keys that provide write access.

  2. credentials are not sufficient - the errorUrl could point to an internally accessible web page that contains other credentials that are more permissive.

Note
 We typically define remotes with full-access credentials in the user’s config file. These can then be referred to when using push and publish command with --remote=user:<remote-name> option. By doing this the write credentials for a remote are not shared with other users. It is however possible that on an internal only project the full-access credentials would be put into a packman level config file or even a project file. There is nothing in the design that prohibits this. Ultimately the choice is yours in how you want to configure the remotes and what level of security is required for the credentials in your project.

Further documentation on the full-access credentials for NVIDIA hosted buckets on S3 can be found on the internal packman wiki:

  1. S3 packman bucket,

  2. S3 cloudfront bucket.

Its possible to store the id and key in system environment variables for automated systems where its not advisable to store them in a configuration file located in source control.

S3 credentials:

  • PM_S3_ID

  • PM_S3_KEY

GTL credentials:

  • PM_GTL_ID

  • PM_GTL_KEY

Note
 Do not provide a <credentials> element for the remote in a config file if you want to pick up the credentials from environment variables.

dependency

A project can contain zero or more dependency elements. The name attribute is required and is used to export the dependency in the user’s environment (after the pull command). The value of the name attribute must therefore be a shell compatible variable name, i.e. contain only alphanumeric characters and underscore. This is verified by the parser. The path to the package or source that fulfills the dependency will be set in the environment variable:

PM_<name>_PATH

And values of all the paths that are produced this way will be concatenated together into a PM_PATHS environment variable as well. This behavior is very convenient for utilities like cmake. These exported paths also follow the Unix convention and use forward slashes.

Optionally a dependency can contain a linkPath attribute. This attribute should contain a relative path to where the dependency will be symbolically linked. The path is relative to the containing project xml file. This allows projects to define dependencies that appear like they are local to the project but still reap the savings of having only a single copy of each dependency in the local packman packages repository. For instance, a project could link all of its dependencies into an external folder at the top of the project source tree under the different dependency names. A project that was previously hosting all of its external dependency packages in source control could also use this mechanism to migrate to packman without having to change any project files by linking the new centrally stored and versioned packman packages to the old source control locations on disk.

Note
 Folder linking requires all users on Windows to host the project locally on an NTFS volume.

With packman 5.14 you can optionally specify a copyPath attribute instead of a linkPath attribute. This attribute should contain a relative path to where the dependency will be copied to. The path is relative to the containing project xml file. This allows projects to define dependencies that are local to the project but has the downside of taking up space through the redundant copy (compared to linkPath approach).

A dependency can contain an additional tags attribute. The value is one or more space separated words that tag the dependency. This can be used in conjunction with the exclude and include tag options in the pull command to include or exclude dependencies from processing. This comes in handy when the same project xml file has to be used in different contexts and the list of dependencies should change depending on context.

A dependency element normally contains at least one Package or Source element.

package

Package elements represent packages that reside on remote package servers. The name and version attributes must be specified. Both of these support variable substitution, i.e. the ${platform} variable can be used to generate the final value of name or version. The value of the version attribute is exported to the environment variable:

PM_<name>_VERSION

This environment variable is typically used by cmake and other similar systems to determine the version of the dependency.

Note
 The value of <name> is taken from the name attribute of the dependency, not the name attribute of the package. This is consistent with the closely related PM_<name>_PATH environment variable (see dependency section above).

An optional platforms attribute can be specified for a package. If left out it means that the package is used by default if an exact match is not found on other packages or sources that specify values for their platforms attribute. For example:

<dependency name="cmake">
   <package name="cmake-x64" version="3.7.0" /> <!--this one is the default -->
   <package name="cmake-linux-x86_64" version="3.7.2" platforms="linux"/>
</dependency>

The cmake-linux-x86_64 will be selected if platform is set to linux with pull command. In all other cases the cmake-x64 package will be chosen (even when no platform is specified with pull command).

It’s important to note that platform is just a name and its meaning is completely up to the user. For some projects it will make sense to use target platforms, like ps4, win, xboxone, as the platform names. In other cases it could be a tools platform name, like linux.make.gcc and mac.make.clang.

A platform name can also include * wild cards, like this:

<dependency name="cmake">
   <package name="cmake-x64" version="3.7.0" platforms="win.*" />
   <package name="cmake-linux-x86_64" version="3.7.2" platforms="linux.* "/>
</dependency>
<dependency name="build_tool">
   <package name="visualstudio2013" version="1.0" platforms="win.vs2013.*" />
   <package name="gnumake-linux" version="2.1" platforms="linux.make.*" />
</dependency>
<dependency name="compiler">
   <package name="clang-linux" version="4.0" platforms="linux.*.clang" />
</dependency>

This would mean that a pull --platform="linux.make.clang" would result in:

  • package "cmake-linux-x86_64" being fetched

  • package "gnumake-linux" being fetched

  • package "clang-linux" being fetched

While a pull --platform="win.vs2013.msvc" would result in:

  • package "cmake-x64" being fetched

  • package "visualstudio2013" being fetched

Note that in the above example wild cards are used for convenience and compatibility with future platforms. The behavior outlined above could also have been achieved with the following project file:

<dependency name="cmake">
   <package name="cmake-x64" version="3.7.0" platforms="win.vs2013.msvc" />
   <package name="cmake-linux-x86_64" version="3.7.2" platforms="linux.make.clang "/>
</dependency>
<dependency name="build_tool">
   <package name="visualstudio2013" version="1.0" platforms="win.vs2013.msvc" />
   <package name="gnumake-linux" version="2.1" platforms="linux.make.clang" />
</dependency>
<dependency name="compiler">
   <package name="clang-linux" version="4.0" platforms="linux.make.clang" />
</dependency>

Even though this is more explicit a problem surfaces when more platform variations are added. For instance, let’s say this project file now needs to support PS4 Orbis toolchain. The file above would need to be expanded like this:

<dependency name="cmake">
   <package name="cmake-x64" version="3.7.0" platforms="win.vs2013.msvc win.vs2013.orbis" />
   <package name="cmake-linux-x86_64" version="3.7.2" platforms="linux.make.clang "/>
</dependency>
<dependency name="build_tool">
   <package name="visualstudio2013" version="1.0" platforms="win.vs2013.msvc win.vs2013.orbis" />
   <package name="gnumake-linux" version="2.1" platforms="linux.make.clang" />
</dependency>
<dependency name="compiler">
   <package name="clang-linux" version="4.0" platforms="linux.make.clang" />
   <package name="orbis" version="3.0" platforms="win.vs2013.orbis" />
</dependency>

Contrast this to the following file using the wild card approach:

<dependency name="cmake">
   <package name="cmake-x64" version="3.7.0" platforms="win.*" />
   <package name="cmake-linux-x86_64" version="3.7.2" platforms="linux.* "/>
</dependency>
<dependency name="build_tool">
   <package name="visualstudio2013" version="1.0" platforms="win.vs2013.*" />
   <package name="gnumake-linux" version="2.1" platforms="linux.make.*" />
</dependency>
<dependency name="compiler">
   <package name="clang-linux" version="4.0" platforms="linux.*.clang" />
   <package name="orbis" version="3.0" platforms="win.*.orbis" />
</dependency>

As you will notice we only needed to make one modification, i.e. adding the Orbis compiler tool-chain for the "compiler" dependency. Which approach you choose is a matter of personal taste and how you foresee your project evolving.

An optional remotes attribute can be specified on a package. This works the same way as in the project element but overrides the general setting provided there. For example, let’s say that we have a package named signtool that needs to remain on our internal network and is therefore only available on NVGTL. The way to specify this is as follows:

<project toolsVersion="5.14" remotes="s3 gtl">
 .....
    <package name="signtool" version="2.1.0.2" remotes="gtl"/>

This would make it so that the signtool package is fetched directly from remote named gtl rather than first trying remote named s3.

label

With packman 5.6 we introduced the concept of a label. In the project file a label element represents labels that reside on the remote servers allowing such things like getting the latest, most stable or specific major releases of a packages for projects that might depend on another packaged product.

The label should be single a line txt file that contains the name and version of the package for packman to fetch. Labels must end with the extension .txt because CDN caching is disabled on those file types. If a label ends with a different extension (.doc for example) then the label will be cached by edge caching servers and changes will take time to replicate between the servers. A time that could be many hours or even days.

Creating a label is easy and only the requires a user publishing a package to create a text file ending with .txt to list the name and version of the package on the first line.

For example: premake@windows.txt could contain premake@5.0.0-master.776a4c9-win.7z.

This is how a project would reference this label:

<project toolsVersion="5.14">
   <dependency name="cmake">
      <label name="premake@windows.txt" />
   </dependency>
</project>

Note how toolsVersion is set to 5.6, to signal that packman 5.6 or newer is needed for this feature. Labels also support the platforms and remotes attributes in the same way a package does. For more information on these attributes please refer to the package section of this help document.

source

This element specifies that the dependency will be fulfilled by a source code folder that is already present locally. The required attribute path needs to designate the path where the source code and built binaries can be found. This path can be specified relative to the project file location. As always the path should be written with forward slashes. The optional attribute platforms can be included and works the same way as for the package element (see above). Likewise the variable ${platform} can be used in the path value. Here is an example:

<source path="googletest/${platform}" platforms="ps4" />

This results in the source and binaries at path googletest/ps4 (relative to project file location) being used to fulfill the dependency for the ps4 platform.
